# 专注小屋 App 改造方案

更新时间：2026-03-21

## 1. 这份文档解决什么问题

这份文档的目标是回答一个很实际的问题：

- 现在这个本地 Flask 网页产品，如果要改造成真正可交付的 App，需要做哪些事。
- 哪条路线最适合当前项目。
- 哪些地方只是“换壳”，哪些地方其实是“重构产品”。

本文默认你说的“App”包含两种可能：

- 安装在电脑上的桌面 App
- 安装在手机上的移动 App

## 2. 当前项目现状

结合当前代码，产品目前更像一个“本地网页工具”，而不是已经接近 App 形态的产品。

- [app.py](/d:/专注小屋/app.py) 是 Flask 入口，既负责页面渲染，也负责接口。
- [static/app.js](/d:/专注小屋/static/app.js) 里已经有实时会话、连续截图、上传截图、展示报告的前端逻辑。
- 当前实时采集依赖浏览器的屏幕共享能力。代码里直接用了 `navigator.mediaDevices.getDisplayMedia`。
- 后端分析能力主要是 Python 本地运行，依赖 `opencv-python`、`pytesseract` 和可选的 DeepSeek 评分。
- 当前项目默认是本地运行，不是云端服务，也没有账号体系、计费、日志、对象存储、版本更新和上架流程。

这意味着：

- 它不是简单“打包一下”就能变成成熟 App。
- 真正要做成产品，需要把“本地网页 + 本地 Python”改造成“客户端 + 可部署后端 + 权限/隐私/发布体系”。

## 3. 最重要的判断

### 判断一：你的当前产品更适合先做桌面 App，而不是立刻做手机 App

原因很直接：

- 现在的核心使用场景是“记录电脑上的专注过程”。
- 当前采集方式依赖浏览器屏幕共享，这本质上是桌面工作流。
- 如果用户主要在电脑上学习、写作、查资料、整理信息，那么桌面 App 和当前产品路径更一致。

### 判断二：如果你坚持要做手机 App，那不是简单迁移，而是产品重定义

因为手机 App 版本会遇到这些变化：

- 采集对象从“电脑屏幕”变成“手机屏幕或 App 内行为”。
- 权限模型完全不同。
- 后台运行、连续截屏、上传、隐私合规都会变复杂。
- iOS 和 Android 的实现方式也不一样。

## 4. 路线选择建议

### 路线 A：先做桌面 App

这是最贴近你当前产品的路线，也是我最推荐的路线。

适合你，如果你的真实目标是：

- 让用户在 Windows 或 macOS 上安装使用
- 继续分析电脑上的学习和工作过程
- 尽量复用现有前端和 Python 分析引擎

这条路线的优点：

- 可以最大化复用现有产品逻辑
- 屏幕采集场景不需要被彻底改写
- OCR 和 Python 分析链条更容易保留
- 从“本地网页工具”升级成“可安装产品”的阻力最小

这条路线需要做的事：

- 选择桌面壳方案，例如 Tauri 或 Electron
- 把当前网页前端嵌入桌面容器，或重做一层桌面 UI
- 处理本地 Python 服务的启动、关闭、日志和异常恢复
- 把 Tesseract、Python 运行时、依赖包一起打包进安装程序
- 做安装器、自动更新、错误日志和版本管理

### 路线 B：先做 Android App

这是“真的做手机 App”的起点路线。

适合你，如果你的目标是：

- 用户主要在手机或平板上使用
- 你接受后端服务上云
- 你愿意重做采集能力和移动端界面

这条路线的特点：

- 可以先做 Android，验证需求和采集可行性
- 技术难度明显高于桌面 App
- 需要新增移动端工程、权限处理、上传队列和云端部署

### 路线 C：Android 和 iOS 同时做

这条路线最重，不建议第一阶段就这样做。

原因：

- Android 和 iOS 的屏幕采集链路不同
- iOS 的实现和审核风险更高
- 你现在的项目还没完成服务化，直接双端同时做，投入会非常高

## 5. 我对当前项目的推荐路线

### 推荐路线

先做“桌面 App 版本”，再决定要不要扩展为手机 App。

### 推荐原因

- 你的当前核心价值不是“网页”本身，而是“连续截图 + OCR + 目标相关评分 + 报告输出”。
- 这套能力天然更贴近电脑端工作流。
- 先做桌面 App，能最快把产品从“本地演示工具”升级成“别人可以安装使用的产品”。
- 如果以后数据证明很多人真的是在手机上使用，再投入 Android 版本会更稳。

## 6. 如果先做桌面 App，需要做什么

### 6.1 产品层

- 明确目标平台：先 Windows，还是 Windows + macOS
- 明确安装方式：单机离线版，还是联网版
- 明确数据策略：截图是否只保存在本机，是否允许上传云端
- 明确授权提示：用户必须知道哪些截图会被处理、保存多久、是否会发到模型服务

### 6.2 架构层

- 拆分“前端展示”和“分析服务”
- 不要再把页面渲染、接口、运行配置全部耦合在一个 Flask 入口里
- 把分析引擎整理成稳定的服务层，保证以后无论是网页、桌面端还是手机端都能复用

### 6.3 桌面端工程

- 选定壳层框架
- 接入本地窗口管理、系统托盘、通知、自动更新
- 处理 App 启动时自动拉起本地分析服务
- 处理 App 关闭时资源释放、缓存清理和异常恢复

### 6.4 本地分析能力打包

- 打包 Python 运行环境
- 打包 OpenCV 和 OCR 依赖
- 打包 Tesseract 和中文语言包
- 提供首次安装后的环境自检

### 6.5 可交付能力

- 制作安装包
- 增加版本号和升级机制
- 增加错误日志导出
- 增加基础埋点，至少知道启动、崩溃、开始记录、结束记录这些关键事件

## 7. 如果要做手机 App，需要做什么

这一部分按“移动 App 版”来写。

### 7.1 先重做产品定义

你要先回答这些问题：

- 你到底要分析什么，是手机自己的屏幕，还是电脑端工作过程
- 用户是以“手机专注”为主，还是“电脑专注”为主
- 实时采集是录屏流、定时截图，还是只分析用户主动上传的图片
- 产品第一版是否必须 iOS 和 Android 同时支持

如果这几个问题不先定，后面技术方案会反复推翻。

### 7.2 拆出云端后端

手机 App 不适合继续依赖“本地 Flask 页面 + 本机 Python 分析”这种结构。

你需要把后端改造成真正的服务：

- 独立 API 服务
- 统一鉴权
- 会话管理
- 文件上传
- OCR 与评分任务处理
- 日志与监控
- 结果持久化

建议保留现有 Python 分析核心，但把它改成可部署服务，而不是只为本地网页服务。

### 7.3 新建移动端客户端

你需要单独新建一个移动端工程，而不是继续在当前模板上直接堆代码。

移动端至少要有这些页面和能力：

- 登录或匿名入口
- 首页
- 创建专注会话
- 实时记录页
- 历史报告列表
- 单次报告详情
- 设置页
- 权限申请页

### 7.4 重做采集链路

这是整个改造里最难的部分之一。

你现在的网页版本依赖浏览器屏幕共享；移动端没有同样简单的一套能力。

Android 侧通常要处理：

- 屏幕采集授权
- 前台服务
- 录屏或帧采样
- 上传队列
- 断网重试
- 电量和发热控制

iOS 侧通常要处理：

- ReplayKit 相关能力
- 录制或广播扩展
- 样本流处理
- 权限提示和用户可感知的录制状态
- 更严格的产品设计和审核边界

### 7.5 重做实时体验

移动端不能只是把网页缩到手机里。

需要重新设计：

- 会话创建流程
- 权限引导
- 录制中的状态页
- 实时反馈方式
- 会话结束后的总结页
- 弱网和中断恢复提示

### 7.6 增加账号和数据体系

如果你想把 App 发给别人用，几乎一定要补这些基础能力：

- 用户账号
- 设备标识
- 会话记录持久化
- 图片或视频对象存储
- 统计报表
- 用户删除数据
- 隐私协议和用户协议

### 7.7 运维与成本控制

一旦做成 App，成本会从“本地运行”变成“持续运营”。

你需要考虑：

- OCR 算力成本
- 模型调用成本
- 图片存储成本
- 带宽成本
- 监控告警
- 崩溃和异常追踪

## 8. 推荐技术方向

这一段是我结合当前项目给出的建议，不是唯一答案。

### 桌面 App 方向

- UI 层可以继续沿用现有网页前端思路
- 容器层优先考虑 Tauri，其次 Electron
- 分析引擎继续保留 Python
- 第一阶段优先做 Windows 版

为什么这样建议：

- 你现在已经有网页界面和 Python 能力，桌面容器能最大化复用
- Tauri 和 Electron 都适合把 Web UI 变成可安装桌面产品
- 你的核心难点在分析链路，不在页面组件本身

### 手机 App 方向

- 客户端建议单独新建跨端工程
- 可选 Flutter，也可选 React Native
- 后端继续保留 Python 分析服务
- 第一阶段优先 Android，不建议一上来就双端同步强推

为什么这样建议：

- Flutter 和 React Native 都适合新建独立移动端工程
- 当前项目的 Python 分析逻辑没必要重写成移动端本地逻辑
- Android 更适合先验证实时采集和上传链路

## 9. 不建议第一阶段做的事

- 不建议第一版就同时做桌面端、Android、iOS 三端
- 不建议第一版就做完全离线、完全本地推理的移动端
- 不建议现在就急着做复杂会员体系和支付
- 不建议在当前 Flask 页面上继续直接堆移动端兼容逻辑
- 不建议没有隐私策略就开始收集用户截图

## 10. 一个现实可执行的阶段计划

### 方案一：桌面 App 优先

第 1 阶段：整理后端和分析引擎

- 拆分配置、接口、分析服务
- 增加统一错误处理
- 增加本地日志
- 做环境自检

第 2 阶段：做 Windows MVP

- 接入桌面壳
- 跑通启动、截图、分析、报告、关闭全链路
- 打包安装程序

第 3 阶段：可交付化

- 自动更新
- 崩溃上报
- 用户设置
- 导出报告

第 4 阶段：再评估是否做手机端

- 看真实用户使用场景
- 决定是否做 Android

### 方案二：手机 App 优先

第 1 阶段：服务化

- 把 Flask 本地工具改成正式 API 服务
- 部署到云端
- 解决上传、存储、日志和鉴权

第 2 阶段：Android MVP

- 完成会话创建、录屏授权、帧上传、实时反馈、历史报告
- 跑通一轮真实使用链路

第 3 阶段：隐私合规和稳定性

- 增加权限说明
- 增加数据删除
- 增加崩溃监控和弱网容错

第 4 阶段：评估 iOS

- 单独评估 ReplayKit 路线、审核风险和产品价值

## 11. 你现在最应该先做的 8 件事

- 明确你要的到底是桌面 App 还是手机 App
- 明确用户主要在哪个设备上完成专注任务
- 明确截图是否允许上传云端
- 把现有分析逻辑整理成稳定服务层
- 把配置、日志、错误处理标准化
- 选定客户端路线：桌面壳、Flutter 或 React Native
- 做一版可运行的 MVP 架构图
- 再开始正式开发新端，而不是继续往当前页面里叠加需求

## 12. 我对你的具体建议

如果你现在是一个人开发，最稳的顺序是：

1. 先把当前项目整理成“可部署后端 + 可复用分析服务”
2. 再做 Windows 桌面 App MVP
3. 用几轮真实测试验证产品价值
4. 最后再判断是否值得做 Android 版

这是因为你的当前能力和产品价值都更靠近“桌面专注记录工具”，而不是“手机录屏分析 App”。

如果你直接跳到手机 App，工作量会一下子从“产品包装和服务化”升级成“产品重构 + 平台权限攻坚 + 运维体系搭建”。
