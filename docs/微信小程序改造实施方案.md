# 专注小屋微信小程序改造实施方案

更新时间：2026-03-21

## 1. 结论先说

- 当前项目不能直接把 Flask 网页前端“换个壳”变成微信小程序。
- 可以高复用的是 `focus_engine` 里的 OCR、目标解析、逐帧分析、报告汇总能力。
- 必须重做的是前端采集链路、会话持久化、账号体系、上传链路和部署方式。
- 最可行的第一阶段路线是：
  原生微信小程序 + 云端 Python API 服务 + 拍照/相册上传 + 专注会话 + 报告复盘。
- 如果你想保留“电脑屏幕连续分析”这个原始价值点，它不应该由小程序单独承担，而应该放到第二阶段，改成“小程序前台 + 桌面采集助手”的双端协同方案。

## 3. 当前项目的真实状态

### 3.1 现在它是什么

当前项目本质上是一个“本地 Flask 网页工具”，不是一个已经服务化、账号化、可发布的小程序后端。

它现在的主链路是：

1. 用户在网页里填写专注目标和时长。
2. 浏览器通过 `getDisplayMedia` 共享屏幕。
3. 前端不断截取画面并上传给 Flask。
4. Python 后端做 OCR、目标相关性评分、会话汇总。
5. 网页端展示实时状态和最终报告。

### 3.2 当前已经具备的核心能力

这些能力可以继续复用：

- 目标输入标准化与关键词扩展
  位置：`focus_engine/goal_profiles.py`
- OCR 图像预处理、缩放、裁剪、缓存
  位置：`focus_engine/ocr.py`
- 单帧分析、去重、并发处理
  位置：`focus_engine/pipeline.py`
- 会话级汇总、建议生成、逐帧得分整理
  位置：`focus_engine/scoring.py`
- 实时会话接口雏形
  位置：`focus_engine/session.py`

### 3.3 当前明显不适合直接搬进小程序的部分

- `templates/index.html`
  这是面向浏览器的一页式页面，不符合小程序的页面组织方式。
- `static/app.js`
  强依赖浏览器 DOM、`fetch`、`FormData`、`ImageCapture`、`navigator.mediaDevices.getDisplayMedia`。
- `app.py`
  同时负责页面渲染和 API，不适合继续作为移动端正式服务入口。
- `StreamingSessionManager`
  现在只把会话放在内存里，适合本地演示，不适合小程序线上使用。

### 3.4 当前项目最大的产品约束

你现在的产品价值，原本更偏向“分析电脑上的工作/学习过程”。

但微信小程序第一阶段最可行的形态，不是“自动抓电脑屏幕”，而是：

- 用户在小程序里发起专注会话
- 用户拍照、从相册选图，或上传已有截图
- 云端完成分析并返回报告

这意味着：如果要做小程序，就必须接受产品定义会发生变化。

## 4. 关于微信小程序边界的关键判断

### 4.1 可以直接利用的小程序能力

结合当前微信小程序公开能力，第一阶段可以可靠使用的能力包括：

- 用户登录：`wx.login`
- 选择媒体：`wx.chooseMedia`
- 相机组件：`camera`
- 相机上下文：`wx.createCameraContext`
- 文件上传：`wx.uploadFile`
- 网络请求：`wx.request`
- WebSocket：`wx.connectSocket`
- 用户隐私保护指引与授权流程

### 4.2 不能按原样照搬的能力

当前网页版本最核心的一步，是浏览器的屏幕共享：

- `navigator.mediaDevices.getDisplayMedia`

这一步在微信小程序里没有当前项目可直接替换的等价链路。

基于当前公开 API 范围，可以得出一个工程判断：

- 微信小程序第一阶段不应设计成“自动连续抓取其他 App 或系统屏幕”
- 你需要把采集方式改成“拍照 / 相册上传 / 小程序前台内的采集流程”

这不是简单改前端，而是产品采集模型要改。

## 5. 我推荐的具体改造路线

### 5.1 推荐路线

先做“纯微信小程序 MVP”，把产品定义改成：

“围绕一个专注目标，用户在小程序里创建会话，通过拍照、相册选图或手动上传资料截图，让系统输出专注相关性报告和复盘建议。”

### 5.2 为什么推荐这条路线

- 它是当前代码能最快落地到微信小程序的一条路。
- 你可以最大化复用 Python 分析引擎。
- 你不需要第一阶段就解决电脑屏幕采集这种跨设备问题。
- 它符合小程序“轻前端、强服务端、明确权限边界”的形态。

### 5.3 不推荐的路线

第一阶段不建议做这些事：

- 不建议尝试把当前 HTML/CSS/JS 原样塞进 WebView
- 不建议在小程序端本地运行 OpenCV、Tesseract、scikit-learn
- 不建议继续把会话状态只存在 Python 进程内存里
- 不建议把“后台自动连续采样”作为第一版目标
- 不建议第一版就同时兼顾微信小程序、H5、桌面端三套前端

## 6. 改造后的目标产品形态

### 6.1 小程序 MVP 的用户流程

建议把第一版流程定义成下面这样：

1. 用户进入小程序首页
2. 登录并进入首页
3. 输入本次专注目标和计划时长
4. 创建一条专注会话
5. 选择采集方式
6. 上传一张或多张图片
7. 后端完成分析
8. 小程序展示实时会话状态和最终报告
9. 用户可以查看历史报告和逐张明细

### 6.2 第一版建议支持的采集方式

建议按优先级分三档：

- P0：相册选图分析
- P0：拍照上传分析
- P1：会话内多次手动补图
- P2：小程序前台内相机连续取帧

说明：

- P0 最稳，最适合先把产品跑通。
- P1 可以保留“会话”概念，但不强求自动采集。
- P2 可以研究，但不应作为第一阶段阻塞项。

## 7. 推荐的技术架构

### 7.1 前端

建议直接使用“原生微信小程序”开发，而不是先上跨端框架。

原因：

- 当前目标平台只有微信小程序
- 相机、媒体选择、上传、隐私授权这类能力，原生小程序接得最稳
- 你现在最缺的是产品闭环，不是多端统一

### 7.2 后端

建议第一阶段继续使用 Python，但把现有 Flask 工程改成“API 服务化”结构。

建议目标形态：

- 保留 `focus_engine/` 作为分析核心
- 将 `app.py` 改造成纯 API 服务入口
- 去掉服务端模板渲染依赖
- 所有返回统一为 JSON

第一阶段可以继续用 Flask，原因是改造成本最低。

等小程序 MVP 跑通后，再考虑是否迁移到 FastAPI。

### 7.3 数据与存储

线上版至少需要新增：

- 用户表
- 专注会话表
- 会话图片表
- 分析报告表
- 文件资源表

建议的存储分层：

- 业务数据：MySQL
- 图片文件：对象存储
- 临时状态与任务队列：Redis

### 7.4 分析执行方式

建议按阶段拆：

- MVP：同步分析
  适合单张图和少量图片上传
- 第二阶段：异步任务队列
  适合批量分析、会话内多图处理、失败重试

## 9. 你需要完成的事情清单

### 9.1 产品与交互

- 明确第一版只做微信小程序，不做桌面端同步重构
- 明确第一版主场景是“拍照/相册上传 + 报告复盘”
- 明确是否保留“专注时长”概念
- 明确是否需要历史报告列表
- 明确是否需要匿名使用，还是必须登录

### 9.2 小程序前端

- 新建原生微信小程序工程
- 搭建全局登录态和请求封装
- 新建首页、创建会话页、上传页、报告页、历史页、我的页面
- 接入 `wx.login`
- 接入 `wx.chooseMedia`
- 接入 `wx.uploadFile`
- 接入隐私授权提示
- 重做所有报告 UI

### 9.3 后端服务

- 从 `app.py` 中移除 HTML 页面渲染
- 重组 API 路由
- 增加统一错误码和响应结构
- 增加用户鉴权
- 增加数据库持久化
- 增加文件上传和对象存储
- 增加历史报告查询接口

### 9.4 会话系统

- 重写 `StreamingSessionManager`
- 支持会话创建、进行中、完成、取消
- 支持一条会话关联多张图片
- 支持会话结果落库
- 支持历史会话再次查看

### 9.5 分析引擎

- 保留 `FocusAnalyzer` 作为核心服务

### 9.6 部署与运维

- 准备 HTTPS 域名

- 准备可部署的 Python 服务环境
- 准备数据库和对象存储

### 9.7 合规与隐私

- 在小程序后台配置用户隐私保护指引
- 对相册、相机、上传用途做明确说明
- 明确截图/照片是否长期保存
- 提供删除记录能力
- 明确模型调用和第三方服务说明

### 9.8 测试与验收

- 真机测试登录
- 真机测试拍照
- 真机测试相册选图
- 真机测试多图上传
- 真机测试报告展示
- 测试断网、超时、重复上传、失败重试

## 10. 建议的小程序页面设计

### 10.1 页面清单

- 首页
  展示最近一次会话、开始新会话入口、历史报告入口
- 创建会话页
  输入目标、选择时长、选择采集方式
- 采集页
  拍照、选图、查看当前会话进度
- 报告页
  展示平均专注分、专注占比、关键词、建议、逐张明细
- 历史列表页
  查看过去会话
- 个人中心页
  隐私说明、缓存管理、反馈入口

### 10.2 交互建议

- 第一版不要做太重的实时动画
- 报告页优先做“总览 + 逐张明细”
- 上传过程要有明确进度提示
- 分析失败时要给出可理解的提示，不要只返回技术错误码

## 11. 建议的 API 设计

### 11.1 新增或改造后的核心接口

- `POST /api/v1/auth/wechat/login`
  使用 `wx.login` 的 code 换取服务端登录态
- `POST /api/v1/sessions`
  创建专注会话
- `GET /api/v1/sessions/{session_id}`
  查看会话状态
- `POST /api/v1/sessions/{session_id}/frames`
  上传单张图片并分析
- `POST /api/v1/sessions/{session_id}/complete`
  完成会话
- `GET /api/v1/reports`
  获取历史报告列表
- `GET /api/v1/reports/{report_id}`
  获取报告详情

### 11.2 当前接口怎么处理

- `GET /`
  废弃
- `POST /analyze`
  废弃 HTML 渲染，只保留 API 形式
- `POST /api/analyze`
  可保留，作为批量分析接口
- `POST /api/session/start`
  可迁移为 `POST /api/v1/sessions`
- `POST /api/session/<session_id>/frame`
  可迁移为 `POST /api/v1/sessions/{session_id}/frames`
- `GET /api/session/<session_id>`
  可迁移为 `GET /api/v1/sessions/{session_id}`
- `POST /api/session/<session_id>/complete`
  可迁移为 `POST /api/v1/sessions/{session_id}/complete`

## 12. 推荐的实施阶段

### 阶段 1：后端服务化

目标：

- 保住分析核心
- 去掉网页渲染耦合
- 建立纯 API 服务

要做的事：

- 重构 `app.py`
- 抽出 API 层和 service 层
- 定义统一响应格式
- 增加数据库模型
- 增加上传存储逻辑

### 阶段 2：小程序 MVP

目标：

- 跑通登录、建会话、上传图片、返回报告

要做的事：

- 新建原生小程序
- 接入登录
- 接入图片选择和上传
- 接入报告页
- 跑通真机链路

### 阶段 3：线上化

目标：

- 让别人可以稳定使用

要做的事：

- 配置正式域名

- 配置对象存储

### 阶段 4：增强版能力

- 批量上传优化

## 14. 第二阶段的扩展建议

如果你后面仍然想保留“电脑专注过程分析”，推荐第二阶段走双端协同：

- 小程序负责登录、创建目标、查看报告、查看历史
- 桌面端小助手或浏览器插件负责抓取电脑截图
- 桌面端直接把图片发到云端 API
- 小程序只作为控制台和报告查看端

这条路更接近你当前项目的原始价值，也比强行让小程序自己抓屏更现实。

## 15. 最终建议

对当前项目来说，最稳的落地方式不是“网页直接改小程序 UI”，而是：

1. 保留 Python 分析引擎
2. 先把后端服务化
3. 新建原生微信小程序前端
4. 把采集方式改成拍照 / 相册上传 / 会话内补图
5. 等 MVP 稳定后，再决定是否做桌面采集助手

如果只做第一版，我建议你把目标收敛成一句话：

“先做一个能在微信里完成目标输入、图片上传、专注分析和报告复盘的小程序。”

这条路线最具体、最可做，也最符合你当前代码的真实基础。
