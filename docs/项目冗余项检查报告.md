# 项目冗余项检查报告

更新时间：2026-03-21

## 1. 检查范围

本次检查基于当前目录下的实际文件、Flask 路由、前端模板与脚本、`focus_engine` 核心模块之间的引用关系完成。

检查目标是找出这几类内容：

- 当前运行链路完全不需要的文件
- 只在演示或开发阶段使用、可按需移除的文件
- 代码中已经预留但当前没有真正生效的参数、字段或接口
- 是否存在表格、导出物、缓存等可清理产物

说明：

- 这次只做审计，没有替你删除任何文件。
- `venv/` 内的第三方依赖没有逐文件展开审计，只按“本地虚拟环境”整体处理。

## 2. 结论摘要

当前项目规模不大，核心运行文件整体比较集中，没有发现大量历史垃圾文件。

本次确认出的重点冗余项主要有 4 类：

- 自动生成物：`__pycache__/`、`focus_engine/__pycache__/`
- 本地环境目录：`venv/`
- 一个未被项目引用的独立密钥文本文件：根目录 `sk-*.txt`
- 一批“已暴露但当前前端没有使用”或“参数已预留但没有真正生效”的代码项

另外：

- 当前目录下没有发现现存的 `.csv`、`.xlsx`、`.xls`、`.json` 这类表格/导出结果文件
- 当前图片文件只有 `测试.png` 这一张，它不是运行主链路必需文件，只服务于演示接口

## 3. 确认可清理项

### 3.1 自动生成缓存

文件/目录：

- `__pycache__/`
- `focus_engine/__pycache__/`

判断：

- 这是 Python 运行后自动生成的字节码缓存，不属于源码资产
- 删除后不会影响项目逻辑，后续运行时会重新生成

建议动作：

- 可以直接删除
- 如果后续纳入版本管理，也不要提交这两个目录

### 3.2 本地虚拟环境

文件/目录：

- `venv/`

判断：

- 这是本机开发环境，不是项目源码的一部分
- 对“在你这台机器上直接运行项目”有帮助，但对代码仓库整理和交付包瘦身没有必要长期保留在项目目录里

建议动作：

- 如果你的目标是整理源码目录、准备备份或发给别人，`venv/` 可以排除或移走
- 如果你当前还在本机继续开发，可以先保留

### 3.3 独立密钥文本文件

文件/目录：

- 根目录 `sk-*.txt`

判断：

- 全项目搜索只发现这个文件本身，没有任何代码读取它
- 当前配置读取的是 `.env` 或系统环境变量，不会读取这个独立文本文件
- 这个文件不但大概率用不到，而且存在凭证明文散落在项目目录的风险

证据：

- `focus_engine/config.py` 只会加载 `.env`，并从环境变量读取 `DEEPSEEK_API_KEY`
- 全项目没有对这个 `sk-*.txt` 的引用

建议动作：

- 优先处理
- 如果它确实是密钥，建议删除该文件，并把密钥只放进 `.env` 或系统环境变量

## 5. 当前未接线或冗余的代码项

### 5.1 当前前端未使用的 API

代码位置：

- `app.py` 中的 `POST /api/analyze`
- `app.py` 中的 `GET /api/analyze-demo`
- `app.py` 中的 `GET /health`

判断：

- 当前内置前端只实际调用了会话相关接口：
  `POST /api/session/start`、`POST /api/session/<session_id>/frame`、`POST /api/session/<session_id>/complete`
- 没有发现当前页面 JS 调用 `/api/analyze`、`/api/analyze-demo` 或 `/health`

建议动作：

- `/api/analyze`：如果你计划以后让外部客户端或脚本直接上传截图，这个接口可以保留
- `/api/analyze-demo`：如果不再需要 demo，建议和 `测试.png` 一起删除
- `/health`：如果没有部署监控需求，可以先保留也可以移除，优先级不高

### 5.2 `runtime_snapshot()` 中未被当前 UI 使用的字段

代码位置：

- `app.py` 的 `runtime_snapshot()`
- `templates/index.html`
- `static/app.js`

当前未被消费的字段：

- `ocr_engine`
- `llm_ready`
- `llm_provider`
- `llm_model`
- `workers`
- `max_uploads`
- `session_window_size`
- `session_capture_mode`
- `default_goal`

判断：

- 这些字段会被后端注入到 `window.focusRuntime`
- 但当前模板和前端脚本实际只使用了 `ocr_ready`、时长配置、支持格式、目标示例等少数字段
- 这些未被消费的字段不会导致错误，但会增加上下文噪音

建议动作：

- 如果你追求接口和模板最小化，可以从 `runtime_snapshot()` 中删掉当前不用的字段
- 如果你准备后续做“运行时状态面板”或调试面板，也可以暂时保留

### 5.3 没有真正生效的配置项

代码位置：

- `focus_engine/config.py` 中的 `session_window_size`
- `focus_engine/config.py` 中的 `enable_temporal_smoothing`

判断：

- `session_window_size` 目前只在 `app.py` 里被塞进 `runtime_snapshot()`，没有进入真实计算逻辑
- `enable_temporal_smoothing` 目前没有搜索到任何消费位置，属于已定义但未接线的配置

建议动作：

- 如果短期不打算开放这些能力，建议删掉或至少补注释说明“预留未启用”
- 如果你本来就打算做可配置时间窗和平滑开关，建议尽快把它们真正接入评分逻辑

### 5.4 预留但未实现的采集间隔参数

代码位置：

- `focus_engine/session.py` 中的 `capture_interval_seconds`

判断：

- 参数已经出现在函数签名和返回 payload 中
- 但 `_validate_settings()` 里直接 `_ = capture_interval_seconds`
- 会话元数据里也固定返回 `None`
- 这说明它目前只是未来扩展位，不是实际功能

建议动作：

- 如果产品已经确定走“连续采集”模式，可以先删掉这个参数
- 如果以后还想支持固定间隔截图，就保留，但建议加注释说明它尚未启用

## 8. 建议的清理顺序

如果你准备做一次实际整理，我建议按这个顺序：

1. 先删自动生成物：`__pycache__/`、`focus_engine/__pycache__/`
2. 处理高风险无引用文件：根目录 `sk-*.txt`
3. 决定是否保留 demo：`测试.png` 与 `/api/analyze-demo`
4. 决定是否保留辅助入口：`run_focus_site.bat`、`.gitignore`
5. 最后再收缩代码里的未接线项：未使用 runtime 字段、`capture_interval_seconds`、未生效配置项

## 9. 一句话结论

这个项目当前真正明显的“可清理冗余”并不多，最值得优先处理的是：

- 两个 `__pycache__` 目录
- `venv/` 是否继续跟随项目目录
- 根目录未被引用的 `sk-*.txt`
- 演示接口与演示图片是否还需要
- 若要继续精简，再处理那些“已经定义但没接线”的配置和接口
